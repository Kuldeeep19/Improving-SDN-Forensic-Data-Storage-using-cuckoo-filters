================================================================
  UPDATE — WHAT WAS ACTUALLY BUILT (post-meeting extensions)
================================================================

The original scope above was the starting point. After the project
meeting, we extended the work so that "traceback to source" is no
longer a redefinition — we actually built a real SDN network and do
real path reconstruction. The two updates below are the additions.

----------------------------------------------------------------
UPDATE 1 — WE BUILT A REAL SDN SOFT-ROUTER NETWORK
----------------------------------------------------------------

Instead of one filter pair sitting offline, we now simulate the full
topology from Varun Sharma's Mininet setup:

    1 controller + 6 soft-switches + 4 hosts

Every soft-switch carries its OWN Bloom Filter and its OWN Cuckoo
Filter (just like every real router does in the Sharma paper).
Packets flow through the network along BFS shortest paths, and every
switch on the path records the packet into both of its filters. Attack
packets travel from host H0 to host H3, so the true attack path is
    H0 -> S0 -> S1 -> S2 -> S3 -> H3

This means traceback is no longer "can I still find the packet in the
filter?" — it is now "starting from the victim, can I reconstruct the
exact switches the attacker's packet travelled through?" — exactly as
Shesha Shila's paper defines it.

----------------------------------------------------------------
UPDATE 2 — THREE TRACEBACK STRATEGIES IMPLEMENTED
----------------------------------------------------------------

We implemented and compared three ways of answering the "who attacked
me?" question:

  (A) Bloom + SPIE query-all
      The controller asks every switch in parallel:
      "did you see this packet?". The YES-switches form the path.
      Cost: O(N) queries. Result: 0/100 — the Bloom wipes destroyed
      the evidence at 14 switches, so every switch replies NO.

  (B) Cuckoo + SPIE query-all
      Same strategy, but with Cuckoo Filters.
      Cost: O(N) queries. Result: 100/100 — every attack path is
      fully recovered.

  (C) Cuckoo + Parent-Pointer walk-back  [the new idea]
      Each switch stores an extra field per packet: the ID of the
      upstream switch that handed it the packet. The controller
      starts at the victim's switch, asks "who sent you this?",
      follows the pointer, asks again, and so on — hop by hop back
      to the source.
      Cost: O(path length) queries (NOT O(N)). Result: 97/100
      successful, 79/100 reconstruct the EXACT ORDERED path.
      Only method that recovers the direction of travel.

----------------------------------------------------------------
UPDATE 3 — SDN CONTROLLER ORCHESTRATION
----------------------------------------------------------------

We added an SDNController class that mirrors the OpenFlow control
plane. When the victim's IDS detects an attack, the excerpt goes to
the controller, and the controller runs all three traceback methods
on the victim's behalf and reports how many switch queries each one
needed. This gives us the query-efficiency comparison.

----------------------------------------------------------------
UPDATE 4 — NETWORK-LEVEL FPR / FNR MEASUREMENT
----------------------------------------------------------------

We now measure not just per-filter FPR but also NETWORK-level
accuracy — how often a switch wrongly answers YES for a packet that
never passed through it (FP), and how often a switch wrongly answers
NO for a packet that did (FN). Results on 100 tracked attacks:

                     Bloom      Cuckoo
    FP (wrong YES)   0.00%      0.00%
    FN (wrong NO)    100.00%    6.50%

The Bloom FN is catastrophic because the wipes erased everything.

----------------------------------------------------------------
UPDATE 5 — SCALABILITY SWEEP
----------------------------------------------------------------

To show the approach is not tied to the 6-switch topology, we rerun
the experiment with N = 6, 10, 20, 50 switches:

    N = 6   -> SPIE  6 queries,  PP 4 queries,  both ~100%
    N = 10  -> SPIE 10 queries,  PP 4 queries,  both ~99-100%
    N = 20  -> SPIE 20 queries,  PP 3 queries,  both ~99-100%
    N = 50  -> SPIE 50 queries,  PP 3 queries,  both ~99-100%

At N=50, parent-pointer uses ~17x fewer controller queries than SPIE
while maintaining the same accuracy. This is the scalability story.
